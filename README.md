# Orbitto Frontend Challenge (React + Reatom + Kubb)

Frontend-реализация auth-флоу Orbitto под backend по OpenAPI-контракту.

Решение покрывает 3 пользовательских сценария:
- регистрация
- авторизация
- восстановление пароля

## Что реализовано

- **Auth-экраны и состояния**: login/register/recovery + success/error/check-email состояния.
- **Приватная зона**: минимальный `dashboard` (`/root/dashboard`) и logout.
- **Route guard**: единая проверка авторизации и редиректы на уровне `AppRouter` (через Reatom routing).
- **Интеграция с backend-контрактом**:
  - типизированный API-клиент и ресурсы, сгенерированные из OpenAPI
  - обработка 401/403 на защищённых запросах
- **Refresh-токены и повтор запроса**:
  - при истёкшем access token выполняется refresh по `refresh_token`
  - исходный запрос повторяется с новым `Authorization`
  - при неуспешном refresh — принудительный logout и редирект на логин
- **MSW для dev**: опциональный режим моков для разработки без живого backend.

## Технологии и почему они выбраны

### React 19
UI-слой приложения. Используется для композиции страниц/форм и предсказуемого рендера. В проекте React дополняется Reatom как инфраструктурным слоем состояния/эффектов.

### Vite
Быстрая сборка и dev server для SPA. В рамках задачи SSR не требуется, поэтому Vite даёт более простой и быстрый пайплайн разработки.

### Reatom v1000 (`@reatom/core`, `@reatom/react`)
State management + routing + эффекты без “размазывания” логики по компонентам.

Ключевые применения:
- **Routing**: `urlAtom` и `RouteAtom` используются как источник истины для текущего URL и матчинга маршрутов.
- **Route guard**: `effect` в `AppRouter` обеспечивает централизованные правила доступа (куда можно гостю, куда нельзя).
- **Токены в cookie**: `withCookieStore` хранит `access_token / refresh_token / expires_in / token_type` в cookies (и синхронизирует с атомами).

### Kubb (OpenAPI codegen)
Генерирует типы/схемы/ресурсы из OpenAPI (`src/shared/__api__/generator/openapi.yaml`):
- `__types__` — TS-типы контрактов
- `__schemas__` — zod-схемы (валидация)
- `__resources__` — типизированные функции запросов (axios-based)
- `__mocks__` — MSW handlers

Почему: меньше ручной типизации, меньше рассинхрона с backend и проще обновлять контракты.

### Axios + `axios-retry`
HTTP-клиент для ресурсов и инфраструктурная надежность:
- retry на сетевые ошибки и 5xx (для отдельного инстанса)
- авторизационные заголовки на request interceptor
- обработка 401/403 и refresh/повтор запроса

### MSW (`msw`)
Опциональная эмуляция API в dev. Нужна, чтобы:
- разрабатывать UI независимо от backend
- воспроизводимо получать ответы/ошибки

### Tailwind CSS v4 + `tailwind-variants`
Tailwind используется как быстрый утилитарный стиль, а `tailwind-variants` (`tv`) — для выноса классов из JSX в `*.variants.ts` (слоты/варианты), чтобы держать стили структурированно.

### HeroUI (`@heroui/react`)
Готовые доступные компоненты (Input/Button и т.п.) — ускоряют реализацию форм и состояний, чтобы сфокусироваться на инженерной части (контракты, устойчивость, архитектура).

### i18n (`i18next`, `react-i18next`)
Тексты вынесены в переводы (например, `public/translate/ru.json`), чтобы:
- не держать строки в компонентах
- подготовить проект к мультиязычности

### Тесты (Vitest + Testing Library)
Инфраструктура unit-тестов через `vitest` и `@testing-library/*` (см. скрипт `test:unit`). Подход подходит для проверки критичных сценариев форм/валидации/UX-состояний.

## Как запустить

### 1) Установить зависимости

Проект использует **pnpm** (зафиксировано через `preinstall`).

```bash
pnpm install
```

### 2) Настроить окружение

В `.env`/`.env.example` используются переменные:
- `VITE_TARGET` — base URL backend (например `http://localhost:8080`)
- `VITE_ENVIRONMENT` — `development | staging | production`
- `VITE_API_MOCKING` — `true/false` (включить MSW)

### 3) Запустить dev-сервер

```bash
pnpm dev:react
```

## Генерация API (OpenAPI → Kubb)

Обновить/перегенерировать клиент:

```bash
pnpm api:generate
```

Если нужно скачать свежий OpenAPI:

```bash
pnpm api:download
```

## Архитектура (FSD-подобная структура)

Код разделён по ответственности:
- `src/app` — инициализация приложения, провайдеры, роутинг
- `src/pages` — страницы (композиция фич/виджетов)
- `src/widgets` — крупные UI-композиции (layout и т.п.)
- `src/features` — сценарии (формы логина/регистрации/восстановления)
- `src/entities` — доменные сущности и маршруты (`ROUTES`)
- `src/shared` — переиспользуемые библиотеки/инфраструктура (api-client, генератор, ui-kit, i18n)

## Trade-offs / что бы сделал дальше

- Добавил бы e2e на живом backend (happy path + негативные сценарии).
- Выделил бы единый слой обработки ошибок API (маппинг кодов → UI-сообщения).
- Усилил бы наблюдаемость (логирование ошибок auth/refresh, метрики конверсии recovery).
